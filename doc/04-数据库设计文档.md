# 朋友组局微信小程序数据库设计文档

## 1. 文档目标

本文件聚焦数据库层设计，重点解决三类问题：

1. 活动、成员、费用、结算的主业务模型。
2. 微信群分享后的邀请、查看、同意、婉拒及原因记录。
3. 用户统计画像和朋友圈熟人排行的数据沉淀。

设计原则：

1. 主实体和行为事件分离。
2. 当前状态和事件流水分离。
3. 高频查询做聚合冗余，避免全表实时计算。
4. 业务可解释，后续可扩展。

## 2. 数据域划分

建议数据库按六个数据域理解：

1. 用户域
2. 活动域
3. 邀请与回执域

## 5. 关键关系说明

### 5.1 activities 与 activity_members

1. 一对多。
2. 一个活动可以有多个成员。

### 5.2 activities 与 invitation_links

1. 一对多。
2. 一个活动可以生成多次分享。

### 5.3 activity_invites 与 invite_events

1. 一对多。
2. 一个邀请主记录可以对应多次状态变化。

### 5.4 activities 与 activity_rsvp_stats

1. 一对一。
2. 用于活动实时回执摘要。

## 6. 为什么要把邀请模型单独拆出来

如果把已查看、已婉拒、已加入都塞进 activity_members，会带来几个问题：

1. 成员表语义被污染。
2. 无法表达一个人看过但未加入。
3. 无法保留婉拒原因和多次状态变化。
4. 发起人实时看板很难做。

因此正确做法是：

1. 成员表只记录真正入局的人。
2. 邀请表记录当前回执状态。
3. 事件表记录所有过程。
4. 聚合表负责实时展示性能。

## 7. 推荐查询场景

### 7.1 发起人查看实时反馈

查询路径建议：

1. 先查 activity_rsvp_stats。
2. 再查 activity_invites 获取已同意、已婉拒、已查看未回复名单。
3. 婉拒原因来自 latest_decline_reason_code 和 latest_decline_reason_text。

### 7.2 用户点击活动详情

处理顺序建议：

1. 写入 activity_view_logs。
2. 写入 invite_events(viewed)。
3. 更新 activity_invites.current_status 为 viewed。
4. 异步刷新 activity_rsvp_stats。

### 7.3 用户点婉拒

处理顺序建议：

1. 写入 invite_events(declined)。
2. 更新 activity_invites.current_status 为 declined。
3. 更新 latest_decline_reason_code 和 latest_decline_reason_text。
4. 刷新 activity_rsvp_stats。

### 7.4 用户点同意入局

处理顺序建议：

1. 写入 invite_events(accepted)。
2. 更新 activity_invites.current_status 为 accepted。
3. 插入或更新 activity_members。
4. 分配 seat_index。
5. 刷新 activity_rsvp_stats。

## 8. 实时性的实现建议

发起人希望实时看到谁同意、谁拒绝、谁进来看过，建议服务端采用以下策略：

1. 写事件时同步更新主状态表。
2. 聚合统计可异步刷新，但延迟应控制在 1 到 3 秒内。
3. 小程序活动详情页每隔数秒拉取一次，或使用 WebSocket/长轮询。
4. 首版如果不做 WebSocket，用短轮询也够用。

## 9. 数据口径建议

### 9.1 viewed_count

1. 同一用户重复进入，只记一次唯一查看人数。
2. 匿名访问不建议直接计入最终发起人名单，避免脏数据。

### 9.2 accepted_count

1. 当前状态为 accepted 且仍有效。
2. 后续若退出活动，可从 accepted_count 中扣除，或单独展示曾同意后退出。

### 9.3 declined_count

1. 当前状态为 declined 的人数。
2. 若后续改为参加，应从 declined_count 中扣除。

### 9.4 pending_count

公式建议：

$$
pending\_count = invited\_count - accepted\_count - declined\_count
$$

## 10. 首版数据库设计结论

对于你这个产品，数据库设计最容易出问题的地方不是费用，而是邀请链路。

真正要稳定支持转发给群里，然后实时看到谁点进来、谁同意、谁婉拒、婉拒原因是什么，必须至少有这 5 张表：

1. invitation_links
2. activity_invites
3. invite_events
4. activity_view_logs
5. activity_rsvp_stats

这 5 张表和活动主表、成员表配合后，整个邀请反馈闭环才是完整的。

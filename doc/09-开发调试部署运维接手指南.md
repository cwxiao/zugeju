目标不是写概念，而是把这个工程现在怎么跑、怎么改、怎么查问题、怎么上线，全部落成一份可执行手册。

## 1. 先说结论：你刚才本地联调为什么“请求没到后端”

你这个问题大概率不是 IDEA 没启动起来，而是请求在 **微信开发者工具侧就被拦住了**，所以根本还没发到 Spring Boot。

这个工程本地联调时，最常见的 5 个原因是：

1. 前端请求地址配错了。
2. 你用的是 `localhost`，但开发者工具/代理场景下更稳的是 `127.0.0.1`。
3. 微信开发者工具没有关闭“合法域名 / TLS / HTTPS 证书”校验。
4. 你不是在开发者工具模拟器里调，而是走了真机预览或真机调试。真机里 `localhost` 指向的是手机自己，不是你的电脑。
5. 后端虽然启动了，但数据库或 Redis 没起来，Spring Boot 实际没完成启动。

这个工程我已经把请求配置改成了：

1. 默认走生产：`https://ywsfs.cn`
2. 本地联调时切换到：`http://127.0.0.1:8080`

对应文件：

1. [frontend/utils/request.js](c:/work/play_minipro/frontend/utils/request.js)

默认这样做是为了避免你以后上传体验版时，前端还误打到本地地址。


你应该看到 `postgres` 和 `redis` 都是 `running` 或 `healthy`。

### 3.2 再启动后端

#### 方式 B：命令行启动


## 4. 本地后端启动时你必须知道的配置点

配置文件：

1. [backend/src/main/resources/application.yml](c:/work/play_minipro/backend/src/main/resources/application.yml)
2. [backend/local-secrets.yml](c:/work/play_minipro/backend/local-secrets.yml)

默认本地配置：

1. 后端端口：`8080`
2. PostgreSQL：`jdbc:postgresql://localhost:5433/play_minipro`
3. Redis：`localhost:6379`
4. JWT Secret：本地默认值已提供

### 4.1 本地登录建议先开 mock

现在你的 [backend/local-secrets.yml](c:/work/play_minipro/backend/local-secrets.yml) 里是：

```yml
app:
  wechat:
    app-secret: 你的真实微信密钥
    mock-login-enabled: false
```

这意味着：

1. 本地登录会走真实微信 `jscode2session`
2. 一旦微信配置、网络、code、AppSecret 有问题，登录就会失败

如果你只是本地做业务联调，我建议你把它改成：

```yml
app:
  wechat:
    mock-login-enabled: true
```

这样本地登录会直接用 mock code 生成稳定 openId，不会被微信真实登录链路拖住。

等你要验证真实登录，再切回：

```yml
app:
  wechat:
    mock-login-enabled: false
    app-secret: 你的真实 AppSecret
```

### 4.2 `local-secrets.yml` 不要继续乱传、乱贴、乱提交

这里面可能放真实微信密钥。后续建议：

1. 保留本地文件，不提交真实值
2. 真正敏感值以后改用环境变量
3. 服务器上只放在 `.env` 或服务器私有配置里

## 5. 本地微信开发者工具怎么和本地后端联调

这是你最关心的部分。

### 5.1 只说正确姿势

你如果要“电脑本地后端 + 微信开发者工具本地前端”联调，推荐流程是：

1. 启动 PostgreSQL + Redis
2. 启动本地后端
3. 浏览器或 curl 验证 `http://127.0.0.1:8080/actuator/health`
4. 用微信开发者工具打开 [frontend](c:/work/play_minipro/frontend)
5. 在开发者工具里切成本地 API 环境
6. 只在模拟器里调，不走真机预览

### 5.2 这个工程现在怎么切本地 API

我已经把请求工具改成“默认生产，可手动切本地”。

文件：

1. [frontend/utils/request.js](c:/work/play_minipro/frontend/utils/request.js)

默认行为：

1. 默认环境：`prod`
2. 生产地址：`https://ywsfs.cn`
3. 本地地址：`http://127.0.0.1:8080`

你在微信开发者工具里，打开调试器 Console，执行：

```js
wx.setStorageSync('apiEnv', 'local')
```

然后重新编译或重新进入小程序页面。

如果你要切回线上：

```js
wx.setStorageSync('apiEnv', 'prod')
```

如果你要临时打某个测试地址，比如局域网 IP：

```js
wx.setStorageSync('customBaseUrl', 'http://192.168.31.20:8080')
```

清掉自定义地址：

```js
wx.removeStorageSync('customBaseUrl')
```

### 5.3 为什么这里用 `127.0.0.1`，不推荐继续用 `localhost`

原因很简单：

1. `127.0.0.1` 更直接，少一层本机 host 解析
2. 某些代理或工具链对 `localhost` 的处理不稳定
3. 你排查网络时，`127.0.0.1` 的可预期性更高

所以以后本地联调统一用：

```text
http://127.0.0.1:8080
```

### 5.4 微信开发者工具必须检查的设置

你当前 [frontend/project.private.config.json](c:/work/play_minipro/frontend/project.private.config.json) 已经有：

```json
"urlCheck": false
```

这说明项目本地配置里已经关闭了 URL 合法域名校验。

但你还是要在微信开发者工具里确认：

1. 详情
2. 本地设置
3. 勾选“不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书”

如果这个没关，`wx.request` 很可能在开发者工具就直接失败，后端日志里什么都看不到。

### 5.5 只能在模拟器里这么联调，真机不行

这点你一定要记住：

1. 在开发者工具模拟器里，`127.0.0.1` 指你的电脑本机
2. 在真机预览、真机调试里，`127.0.0.1` / `localhost` 指手机自己

所以：

1. 如果你要调本地后端，就用开发者工具模拟器
2. 如果你要真机调试，就必须把后端暴露成手机能访问的地址

真机调试可选方案：

1. 用公网测试域名 + HTTPS + 小程序后台合法域名
2. 用云服务器测试环境
3. 用内网穿透，但最终仍然要符合微信合法域名规则

## 6. 本地联调排查清单

以后只要出现“前端请求没到后端”，就按这个顺序排，不要跳步。

### 第 1 步：后端到底有没有起来

```powershell
curl.exe -sS http://127.0.0.1:8080/actuator/health
```

不通就说明后端没起来，先看 IDEA 控制台或命令行日志。

### 第 2 步：数据库和 Redis 到底在不在

```powershell
Set-Location C:\work\play_minipro\backend
docker compose ps
```

### 第 3 步：前端现在到底打的是哪个地址

在微信开发者工具 Console 执行：

```js
wx.getStorageSync('apiEnv')
wx.getStorageSync('customBaseUrl')
```

如果：

1. `customBaseUrl` 有值，就优先走它
2. 否则看 `apiEnv`
3. 不设置时默认走生产 `https://ywsfs.cn`

### 第 4 步：开发者工具有没有拦请求

看微信开发者工具 Network 面板：

1. 如果请求根本不出现，多半是页面逻辑没触发
2. 如果请求出现但立刻报合法域名/TLS 相关错误，是工具侧拦了
3. 如果请求出现并发出，但后端没日志，再看是不是地址不是 `127.0.0.1:8080`

### 第 5 步：后端启动了但登录失败

这时通常不是“请求没到后端”，而是“请求到了但业务失败”。

优先排查：

1. 本地 mock 登录是不是开着
2. 真实微信 `AppSecret` 对不对
3. 小程序 `appid` 对不对
4. Redis 是否正常

## 7. 日常开发建议：先改哪边，后改哪边

### 7.1 改后端时

建议顺序：

1. 先改 DTO / Mapper / Service / Controller
2. 先 `mvn -q -DskipTests compile`
3. 编译过了再开前端联调

### 7.2 改前端时

建议顺序：

1. 先确认接口返回结构
2. 再改页面渲染
3. 再改交互文案和样式

### 7.3 最省 token 的做法

你既然要自己继续开发，又考虑 Copilot 按 token 计费，最省的方式是：

1. 先自己定位到具体文件和具体函数，再问 Copilot，不要让它全仓库大范围探索。
2. 先用本地编译、日志、微信开发者工具 Network 面板把问题缩小，再提问。
3. 一次只让 Copilot 做一件事，比如“补这个接口”或“修这个页面”，不要把多个不相关问题混在一个提示里。
4. 让 Copilot 输出最终文档、脚本、命令模板，这类一次性产出最值。
5. 对已稳定的工程结构、部署命令、排查步骤，优先沉淀成文档，后面直接复用，不要重复问。

## 11. 代码工程上的几个硬规则

以后你自己接着写时，建议守住这些规则，不然很快会乱。

### 11.1 后端

1. Controller 只做入参接收和返回，不堆业务。
2. 业务逻辑放 Service。
3. SQL 查询放 Mapper，别把 SQL 拼到 Service 里。
4. 接口出参统一用 DTO，不要直接把 Entity 暴露给前端。
5. 新增表结构时一定补 Flyway migration。

### 11.2 前端

1. 页面状态放 page `data`。
2. 请求统一走 [frontend/utils/request.js](c:/work/play_minipro/frontend/utils/request.js)。
3. 不要每个页面各自写 base URL。
4. 有环境切换需求时，只改 `request.js`，不要全项目手改接口域名。
5. 真正会复用的格式化和数据映射，再抽到 `utils`。

## 12. 你后面最容易踩的坑

### 12.1 本地接口不通，但线上通

通常是：

1. 开发者工具没关域名校验
2. 前端没切到 `local`
3. 你在真机里访问了 `localhost`

### 12.2 后端启动报数据库错误

通常是：

1. PostgreSQL 没起
2. 端口 `5433` 被占了
3. 本地数据卷坏了

先查：

```powershell
docker compose ps
docker compose logs postgres
```

### 12.3 登录失败

通常是：

1. mock 登录没开
2. 真实微信配置不对
3. Redis 有问题
4. token 已过期但前端没清干净

### 12.4 线上更新了代码但没生效

通常是：

1. 你没重新 `mvn package`
2. 新 JAR 没上传成功
3. 服务器启动没加 `--build`
4. Nginx 还在指向旧服务

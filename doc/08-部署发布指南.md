# 来整部署发布指南

本文档用于指导后续代码更新后的发布操作，分为两部分：

1. 后端更新后如何部署到服务器
2. 前端更新后如何通过微信开发者工具上传体验版/正式版

## 一、当前线上环境

当前项目已经按以下方式部署：

1. 服务器：`49.232.163.23`
2. SSH 用户：`ubuntu`
3. 服务器项目目录：`~/play-minipro-deploy`
4. 线上域名：`https://ywsfs.cn`
5. Nginx 已配置将：
   - `https://ywsfs.cn/api/*` 转发到后端 `127.0.0.1:8080`
   - `https://ywsfs.cn/actuator/*` 转发到后端 `127.0.0.1:8080`
6. 小程序后台已配置合法域名：`https://ywsfs.cn`

当前后端发布不是在服务器上跑 Maven 源码构建，而是：

1. 本地先执行 `mvn package`
2. 生成 `backend/target/play-minipro-backend-0.0.1-SNAPSHOT.jar`
3. 服务器 Docker 镜像直接复制这个 JAR 启动

这意味着：

1. 只改 Java 源码但不重新打包，服务器不会拿到最新代码
2. 每次后端代码更新后，都必须先在本地重新构建 JAR

## 二、后端发布流程

### 1. 本地修改后端代码

后端代码目录：`backend/`

如果修改了以下任一内容，都要重新发布后端：

1. `backend/src/main/java/**`
2. `backend/src/main/resources/**`
3. `backend/Dockerfile`
4. `deploy/docker-compose.prod.yml`
5. 生产环境变量依赖的配置逻辑

### 2. 本地重新打包后端

在项目根目录执行：

```powershell
Set-Location C:\work\play_minipro\backend
mvn -q -DskipTests package
```

打包完成后，确认文件存在：

```powershell
Get-Item C:\work\play_minipro\backend\target\play-minipro-backend-0.0.1-SNAPSHOT.jar
```

### 3. 上传最新后端文件到服务器

#### 只更新业务代码时

通常只需要上传最新 JAR：

```powershell
$key = Resolve-Path 'C:\work\play_minipro\服务器资源\portal_web.pem'
scp -i $key C:\work\play_minipro\backend\target\play-minipro-backend-0.0.1-SNAPSHOT.jar ubuntu@49.232.163.23:~/play-minipro-deploy/backend/target/play-minipro-backend-0.0.1-SNAPSHOT.jar
```

#### 同时修改了部署文件时

如果改了以下内容：

1. `backend/Dockerfile`
2. `deploy/docker-compose.prod.yml`
3. `deploy/.env.example`

则需要把对应文件也一起传上去，例如：

```powershell
$key = Resolve-Path 'C:\work\play_minipro\服务器资源\portal_web.pem'
scp -i $key C:\work\play_minipro\backend\Dockerfile ubuntu@49.232.163.23:~/play-minipro-deploy/backend/Dockerfile
scp -i $key C:\work\play_minipro\deploy\docker-compose.prod.yml ubuntu@49.232.163.23:~/play-minipro-deploy/deploy/docker-compose.prod.yml
scp -i $key C:\work\play_minipro\deploy\.env.example ubuntu@49.232.163.23:~/play-minipro-deploy/deploy/.env.example
```

### 4. 登录服务器

```powershell
$key = Resolve-Path 'C:\work\play_minipro\服务器资源\portal_web.pem'
ssh -i $key ubuntu@49.232.163.23
```

### 5. 检查生产环境变量

服务器目录：`~/play-minipro-deploy/deploy`

进入目录：

```bash
cd ~/play-minipro-deploy/deploy
```

查看当前 `.env`：

```bash
cat .env
```

当前线上至少应包含这些关键变量：

```text
POSTGRES_DB=play_minipro
POSTGRES_USER=play
POSTGRES_PASSWORD=...
JWT_SECRET=...
WECHAT_MINI_APP_ID=wxb8fd5841e5f0a83d
WECHAT_MINI_APP_SECRET=你的微信小程序AppSecret
WECHAT_MOCK_LOGIN_ENABLED=false
```

如果是正式微信登录环境，必须确保：

1. `WECHAT_MINI_APP_SECRET` 已填写
2. `WECHAT_MOCK_LOGIN_ENABLED=false`

### 6. 重建并重启后端容器

在服务器执行：

```bash
cd ~/play-minipro-deploy/deploy
docker compose --env-file .env -f docker-compose.prod.yml up -d --build --force-recreate backend
```

说明：

1. `--build` 会重新构建镜像，确保新 JAR 被打进镜像
2. 如果少了 `--build`，即使上传了新 JAR，也可能仍然跑旧镜像

### 7. 检查后端是否启动成功

在服务器执行：

```bash
cd ~/play-minipro-deploy/deploy
docker compose --env-file .env -f docker-compose.prod.yml ps
docker logs --tail 100 laizheng-backend
curl http://127.0.0.1:8080/actuator/health
```

健康检查返回示例：

```json
{"status":"UP"}
```

### 8. 检查公网是否正常

本地机器执行：

```powershell
curl.exe -sS https://ywsfs.cn/actuator/health
```

如果需要验证登录接口链路，可用真实小程序 `wx.login()` 拿到的 `code` 调用：

```powershell
$body = @{ code = '真实code'; nickname = 'test'; avatarUrl = '' } | ConvertTo-Json -Compress
Invoke-RestMethod -Uri 'https://ywsfs.cn/api/auth/wechat-login' -Method Post -ContentType 'application/json' -Body $body
```

## 三、前端发布流程

### 1. 本地修改前端代码

前端目录：`frontend/`

当前前端线上接口地址已经固定为：

```text
https://ywsfs.cn
```

对应文件：`frontend/utils/request.js`

上传体验版前，确认这里不要再改回本地 IP 或测试地址。

### 2. 用微信开发者工具打开前端项目

使用微信开发者工具打开：

```text
C:\work\play_minipro\frontend
```

确认项目 AppID 为：

```text
wxb8fd5841e5f0a83d
```

### 3. 本地联调检查

上传前建议至少检查：

1. 首页能正常打开
2. 登录流程能触发 `wx.login`
3. 页面请求走的是 `https://ywsfs.cn/api/...`
4. 不存在写死的局域网地址、`localhost`、测试地址

### 4. 上传体验版

在微信开发者工具中：

1. 点击“上传”
2. 填写版本号，例如：`0.1.3`
3. 填写版本备注，例如：`修复真实微信登录并更新活动接口`

上传完成后，在微信公众平台的小程序后台：

1. 进入“版本管理”
2. 找到刚上传的体验版
3. 添加体验成员或直接扫码体验

### 5. 体验版验证建议

体验版至少验证以下内容：

1. 首次进入能正常登录
2. 首页能正常拉取“我的进行中活动”
3. 创建活动接口能成功提交
4. 活动详情能正常打开
5. 账单页接口能正常返回

如果体验版登录失败，优先检查：

1. 小程序后台合法域名是否仍包含 `https://ywsfs.cn`
2. 后端 `WECHAT_MINI_APP_SECRET` 是否正确
3. 后端 `WECHAT_MOCK_LOGIN_ENABLED` 是否误开
4. 前端是否仍指向 `https://ywsfs.cn`

## 四、推荐发布顺序

以后每次发版，建议按这个顺序操作：

1. 本地改后端代码
2. 本地 `mvn -q -DskipTests package`
3. 上传 JAR 和必要的部署文件到服务器
4. 服务器执行 `docker compose --env-file .env -f docker-compose.prod.yml up -d --build --force-recreate backend`
5. 验证 `https://ywsfs.cn/actuator/health`
6. 本地改前端代码
7. 微信开发者工具打开 `frontend/` 目录联调
8. 上传体验版
9. 真机扫码验证核心流程

## 五、常见问题

### 1. 上传了新 JAR，但线上还是旧逻辑

原因通常是只重启了容器，没有重建镜像。

正确命令：

```bash
docker compose --env-file .env -f docker-compose.prod.yml up -d --build --force-recreate backend
```

### 2. 后端健康检查正常，但小程序登录失败

常见原因：

1. 前端没有重新登录，还在使用旧 token
2. 小程序后台合法域名没配置对
3. `AppSecret` 错误
4. 微信返回的 `code` 已失效或不是当前小程序 AppID 生成的

### 3. 本地调试正常，体验版不正常

这是因为开发者工具里可能关闭了域名校验，但真机体验版会严格校验：

1. HTTPS
2. 合法域名
3. 真实微信登录

所以最终一定要以真机体验版结果为准。

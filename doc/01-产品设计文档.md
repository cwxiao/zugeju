# 朋友组局微信小程序产品设计文档

## 1. 产品概述

### 1.1 产品定位

这是一个面向熟人朋友场景的微信小程序，核心能力是把“发起活动、组人、确认时间地点、记录费用、自动结算”串成一条完整链路。

它不是泛社交产品，也不是单纯的活动报名工具，而是一个专门解决朋友组局效率和费用公平问题的产品。

一句话定义：

一个帮助朋友高效组局、透明记账、公平结算的微信小程序。

### 1.2 产品价值

当前朋友之间约活动，通常依赖微信群沟通，存在以下问题：

1. 发起活动时信息分散，活动内容、时间、地点经常反复确认。
2. 参与人数不直观，不知道还差几个人，也不知道谁真正确定参加。
3. 线下活动经常涉及多人垫付，活动结束后很难公平结算。
4. 活动过程中持续产生费用，容易遗漏，最终扯皮。
5. 一些活动既可线上也可线下，但组织流程没有统一模板。

本产品希望解决这五个问题：

1. 让发起活动变成标准化流程。
2. 让参与人清晰知道活动规则和安排。
3. 让人数、席位、报名状态一眼可见。
4. 让费用记录可追溯、可核对、可自动结算。
5. 对简单约局和复杂出游都能覆盖。
6. 让每个用户的组局行为形成可展示的个人数据画像与社交排行。

### 1.3 目标用户

1. 经常组织朋友活动的人。
2. 小圈子固定搭子群体。
3. 有 AA 需求的聚餐、露营、出游、开黑活动发起人。
4. 希望减少沟通成本、避免费用纠纷的熟人社交用户。

## 2. 产品目标

### 2.1 核心目标

1. 将活动发起时间从群聊反复沟通压缩到 1 分钟内完成配置。
2. 将活动信息标准化展示，减少重复问答。
3. 将费用记录从活动后回忆改为活动中实时补录。
4. 将 AA 结算从人工计算改为系统自动完成。
5. 将用户的组局、请客、AA、参与等行为沉淀为长期统计资产。

### 2.2 成功标准

1. 用户可以快速创建一场线上或线下活动。
2. 参与成员能顺畅加入、占位、查看活动信息。
3. 活动中新增的费用能持续记录并进入结算。
4. 活动结束后系统能给出清晰可信的结算结果。
5. 用户能在个人页和排行榜页查看自己的组局风格与社交活跃度。

## 3. 适用场景

### 3.1 轻量场景

1. 今晚线上开黑。
2. 明天下班聚餐。
3. 周末临时打一场羽毛球。
4. 纯预约型活动，不涉及费用。

### 3.2 中度场景

1. KTV 局，先定人数，费用后续补充。
2. 电竞酒店开黑，涉及房费、零食、打车费。
3. 聚餐后续追加酒水、零食、场地费用。

### 3.3 重度场景

1. 露营，两天内持续产生物资、租车、餐饮、营地费。
2. 钓鱼、自驾、短途旅行，多人分散垫付。
3. 活动中临时加人、有人提前离场、有人只参与部分环节。

## 5. 核心流程

### 5.1 发起人主流程

1. 进入首页。
2. 点击发起活动。
3. 选择活动类型。
4. 系统自动判断活动模式。
5. 设置标题与基础说明。
6. 设置参与人数。
7. 自动生成头像席位。
8. 设置开始时间。
9. 若为线下活动，填写集合地址与活动地址。
10. 选择是否涉及费用。
11. 若涉及费用，选择费用制度。
12. 发布并分享活动。
13. 转发到微信群或好友。
14. 实时查看谁点进来、谁同意入局、谁婉拒。
15. 如有人婉拒，可查看婉拒原因。
16. 活动过程中记录费用。
17. 活动结束后发起结算。

### 5.2 参与人流程

1. 从分享卡片进入活动详情。
2. 系统记录已查看活动。
3. 查看活动类型、时间、地点、费用规则和席位情况。
4. 选择同意入局或婉拒。
5. 婉拒时可选择原因或补充说明。
6. 点击加入活动并占位。
7. 如有费用，可在活动中新增自己的支出。
8. 活动结束后查看结算结果。
9. 根据结果完成付款或收款确认。

## 7. 活动发起功能设计

### 7.1 基础信息

发起活动时需填写以下内容：

1. 活动类型。
2. 活动标题。
3. 活动简介。
4. 活动封面，可选。
5. 发起人信息自动带入。

系统可根据活动类型自动生成默认标题：

1. 今晚 5 人开黑局。
2. 周六露营局。
3. 周五 KTV 局。
4. 周末烧烤搭子局。

### 7.2 活动模式

1. 仅线下活动自动锁定。
2. 线上或线下活动需发起人选择本次形式。
3. 线上活动需补充参与方式。
4. 线下活动需填写地点信息。

### 7.3 人数与席位

1. 设置目标人数。
2. 设置人数上限。
3. 支持候补机制。
4. 支持临时加人。
5. 自动生成头像席位图。

席位展示规则：

1. 已加入成员显示头像。
2. 空位显示占位头像。
3. 候补成员显示在席位外。
4. 发起人有特殊标识。

### 7.3.1 邀请与回执状态

活动发布后，发起人应能轻松转发给微信群，并实时看到邀请反馈。

系统需要区分以下状态：

1. 已送达但未查看。
2. 已查看但未表态。
3. 已同意入局。
4. 已婉拒。

发起人应能看到：

1. 哪些人点进来看过。
2. 哪些人明确同意。
3. 哪些人点击了婉拒。
4. 婉拒的原因是什么。
5. 当前还差几个人成局。

婉拒原因建议支持两种方式：

1. 快速选择。
2. 自定义补充。

常见原因建议预设：

1. 时间不合适。
2. 地点太远。
3. 人数太多或太少。
4. 对这个活动不感兴趣。
5. 费用不方便。
6. 临时有事。
7. 下次一定。
8. 自定义原因。

### 7.3.2 实时成局看板

活动详情页中，发起人应看到一个实时看板：

1. 已查看人数。
2. 已同意人数。
3. 已婉拒人数。
4. 待回复人数。
5. 当前席位占用情况。

这个看板的价值：

1. 发起人不需要回微信群逐条统计。
2. 能快速判断这场局是否能成。
3. 能针对已查看未回复的人做提醒。

### 7.4 时间设置

1. 活动开始时间必填。
2. 活动结束时间选填。
3. 集合时间选填。
4. 支持全天活动。
5. 支持开始前提醒。

规则说明：

1. 无论线上或线下，开始时间都必须填写。
2. 线下活动可额外填写集合时间。
3. 支持跨天活动。

### 7.5 地址设置

线下活动建议拆为两个字段：

1. 集合地址。
2. 活动地址。

原因：

1. 集合地点和活动地点通常不同。
2. 露营、自驾、钓鱼、聚餐转场等场景中很常见。
3. 可减少活动前反复确认。

补充字段：

1. 是否同一地点。
2. 导航链接。
3. 联系电话。
4. 备注说明。

线上活动建议字段：

1. 线上平台。
2. 房间号/群号。
3. 加入方式说明。
4. 活动前提醒文案。

## 8. 费用体系设计

### 8.1 是否涉及费用

发起时让用户选择：

1. 不涉及费用。
2. 可能涉及费用。
3. 一定涉及费用。

产品意义：

1. 不涉及费用时，流程极简化。
2. 可能涉及费用时，允许活动中补录。
3. 一定涉及费用时，提前明确费用规则。

### 8.2 费用制度类型

首版建议支持三种核心模式：

1. AA 制。
2. 我请客。
3. 指定某人请客。

后续扩展模式：

1. 部分 AA，部分请客。
2. 先记账，活动后决定如何分摊。

### 8.3 三种制度定义

1. AA 制：所有参与结算的成员共同分摊相关费用，垫付不影响最终公平性。
2. 我请客：发起人承担全部费用。
3. 指定某人请客：由某一指定成员承担全部费用。

### 8.4 费用记录能力

每条费用记录建议包含：

1. 费用名称。
2. 费用类别。
3. 金额。
4. 付款人。
5. 付款时间。
6. 费用说明。
7. 上传凭证，可选。
8. 分摊范围。
9. 是否纳入结算。
10. 备注。

费用类别建议：

1. 住宿
2. 场地
3. 门票
4. 餐饮
5. 酒水
6. 零食
7. 交通
8. 打车
9. 油费
10. 停车费
11. 装备租赁
12. 游戏充值/开房
13. 物资采购
14. 其他

### 8.5 记账权限

建议规则：

1. 发起人默认可记账。
2. 已加入成员可新增费用。
3. 发起人可关闭“成员自行记账”。
4. 成员新增费用可设置为待确认。
5. 发起人可编辑、驳回、删除异常费用。

## 9. AA 结算设计

### 9.1 结算目标

系统应自动算出：

1. 每个人总共支付了多少。
2. 每个人应承担多少。
3. 每个人最终应收还是应付。

核心公式：

$$
个人净额 = 个人应承担费用 - 个人已支付费用
$$

说明：

1. 结果大于 0，表示该成员还要付款。
2. 结果小于 0，表示该成员应该收款。
3. 所有成员净额之和应为 0。

### 9.2 典型 AA 示例

场景：

1. 活动共 4 人参与。
2. A 支付酒店 500 元。
3. B 支付烧烤 150 元。
4. 总费用为 650 元。
5. 每人应承担 162.5 元。

结果：

1. A 已支付 500，应收 337.5。
2. B 已支付 150，应付 12.5。
3. C 已支付 0，应付 162.5。
4. D 已支付 0，应付 162.5。

### 9.3 结算成员范围

为适配真实场景，需要区分三类人：

1. 报名成员。
2. 实际参与成员。
3. 参与结算成员。

建议产品规则：

1. 默认已加入且未退出的人为结算成员。
2. 发起人可在结算前调整最终结算成员。
3. 单条费用可只针对部分成员分摊。

### 9.4 单条费用定向分摊

典型场景：

1. 酒店费只给过夜成员分摊。
2. 烧烤费给全部成员分摊。
3. 租车费只给乘车成员分摊。

因此每条费用都需要支持“分摊范围”字段。

### 9.5 转账建议

结算完成后，系统应生成最少转账路径，降低操作成本。

示例：

1. A 应收 300。
2. B 应收 100。
3. C 应付 200。
4. D 应付 200。

建议转账：

1. C 转给 A 200。
2. D 转给 A 100。
3. D 转给 B 100。

转账策略：

1. 尽量减少转账笔数。
2. 优先匹配大额应收与大额应付。
3. 每个人只看到自己需要转给谁、收谁的钱。

### 9.6 小数处理

建议规则：

1. 结算精确到分。
2. 分摊时按分四舍五入。
3. 若存在 0.01 元尾差，由系统自动补差。
4. 结算页面展示补差说明，保证账目透明。

## 10. 非 AA 模式设计

### 10.1 我请客

1. 所有费用由发起人承担。
2. 成员只看明细，不参与分摊。
3. 若成员垫付，则由发起人补给该成员。

### 10.2 指定某人请客

1. 发起人指定承担人。
2. 其余逻辑与“我请客”一致。
3. 适合生日局、庆祝局、请客局。

### 10.3 不涉及费用

1. 不显示费用入口，或默认收起。
2. 无需进入记账与结算流程。
3. 保持流程轻量，专注活动组织。

## 13. 边界场景

### 13.1 人员边界

1. 有人退出活动。
2. 有人临时加入。
3. 活动已满后有人候补。
4. 有人报名但没到场。
5. 有人只参加后半场。
6. 有人带朋友临时空降。
7. 有人点进来看了但一直不回复。
8. 有人先婉拒，后来又改成参加。

产品应对：

1. 支持发起人手动调整最终结算成员。
2. 支持单条费用定向分摊。
3. 支持释放席位与候补转正。
4. 支持 RSVP 状态更新，以最后一次有效表态为准。
5. 保留查看记录与婉拒记录，便于活动分析。

### 13.2 时间边界

1. 活动跨天。
2. 时间临时变更。
3. 结束时间不确定。
4. 集合时间与开始时间不一致。

产品应对：

1. 开始时间必填。
2. 结束时间选填。
3. 时间修改后通知参与人。

### 13.3 地址边界

1. 集合地址和活动地址相同。
2. 活动地点临时变更。
3. 地址还未完全确定。
4. 活动多次转场。

首版建议：

1. 支持一个集合地址和一个活动地址。
2. 支持后续编辑更新。
3. 多转场留待后续版本。

### 13.4 费用边界

1. 费用录错。
2. 费用有争议。
3. 有退款。
4. 有优惠券或团购抵扣。
5. 某笔费用并非所有人参与。
6. 有人已线下先行转账。

首版建议：

1. 支持费用编辑与删除。
2. 支持费用待确认状态。
3. 支持退款作为冲销记录处理。
4. 支持标记线下已结清。

## 16. 用户统计与排行榜设计

### 16.1 统计目标

系统不仅要服务单次活动，还要持续沉淀每个微信用户的组局数据，形成个人画像和熟人社交排行。

要记录的核心数据包括：

1. 发起过多少次活动。
2. 参与过多少次活动。
3. 最常发起的活动类型是什么。
4. 请客了多少次。
5. AA 了多少次。
6. 线上局和线下局分别参加多少次。
7. 总垫付金额、总应付金额、总应收金额。
8. 常一起组局的人有哪些。

### 16.2 个人画像卡设计

建议在“我的数据”页面展示用户画像卡，形成轻社交传播感。

画像卡建议字段：

1. 累计发起局数。
2. 累计参加局数。
3. 最常发起活动类型 Top 1。
4. 最常参加活动类型 Top 3。
5. 请客次数。
6. AA 次数。
7. 累计垫付金额。
8. 累计应收回金额。
9. 累计已结清活动数。

示例文案：

1. 你今年已经发起了 28 次局。
2. 你最爱发起的是游戏开黑。
3. 你请客了 6 次，是局里的大气担当。
4. 你参加了 43 次局，属于高频搭子。

### 16.3 排行榜设计

排行榜应定位为“朋友圈熟人排行”，不是全站陌生人排行。

首版可基于活动中真实共同参与过的人形成轻关系圈，展示熟人维度排行。

建议榜单：

1. 发起王榜：谁发起活动最多。
2. 组局劳模榜：谁参加活动最多。
3. 请客榜：谁请客次数最多。
4. AA 榜：谁参与 AA 次数最多。
5. 开黑王榜：谁最常发起游戏局。
6. 聚会王榜：谁最常发起线下吃喝玩乐局。

### 16.4 排行榜范围规则

为避免全平台陌生社交感过强，建议按以下范围生成：

1. 默认展示与我共同参加过活动的熟人圈排行。
2. 可增加“近 30 天”“累计历史”两个时间范围。
3. 首版不做全站排行榜。
4. 对未授权或无共同关系的用户，不展示敏感排行。

### 16.5 排行榜价值

1. 提升社交趣味性和产品留存。
2. 强化用户对自己组局习惯的认知。
3. 让产品从工具属性升级为带轻关系网络的数据产品。
4. 为后续年度总结、组局报告、朋友默契榜打基础。

## 22. 登录与合规要求

### 22.1 微信一键登录

首版必须支持微信一键登录，作为所有活动发起、参与、记账、排行榜统计的身份基础。

登录设计要求：

1. 用户首次进入时，支持微信快捷登录。
2. 获取微信基础身份信息后，创建平台用户。
3. 所有活动、费用、结算、统计数据都绑定到用户唯一身份。
4. 未登录用户可浏览部分内容，但不能发起、参加、记账、结算。

### 22.2 服务端部署要求

由于你已有备案服务器，产品形态按独立后端设计，不依赖微信云开发作为核心架构。

要求：

1. 小程序前端通过 HTTPS 与自有服务端通信。
2. 服务端负责微信登录态校验、业务数据存储和排行榜统计。
3. 所有排行榜、用户画像、结算结果都由服务端统一生成。
4. 为后续正式上线预留域名、证书、日志和风控能力。

## 23. 产品结论

这款产品最有竞争力的点，不是单纯能发起活动，而是三件事同时成立：

1. 发起足够快。
2. 占位足够直观。
3. 结算足够公平。

因此产品设计应始终坚持两个原则：

1. 对不涉及费用的活动，流程必须极简。
2. 对涉及多人垫付的活动，结算必须真实、透明、可追溯。

建议 MVP 产品定义为：

一个专门为朋友组局设计的微信小程序。发起人可以快速创建线上或线下活动，设置人数、时间、地址和费用规则；参与人加入占位；活动中持续记账；活动结束后一键生成公平结算结果。

# 202606012206 改进需求工程设计与调试说明

## 1. 本次改进范围

本次改进覆盖三个用户入口：活动档案、AI 人格档案、首页创建入口。

- 活动档案：从“卡片底部弹窗”改为“独立档案详情页”，详情页同时展示活动基础信息、成员、账单总览、每一笔支出和结算追溯。
- 活动档案筛选：增加状态筛选、活动开始/结束日期筛选，保留身份筛选、关键词搜索和时间排序。
- AI 人格档案：补充六维热力雷达图、请客次数、总花费、AA 花费、请客花费，并支持日/周/月/季/年财报视图。
- 首页：主按钮文案从“整一下”改为“整!”，首页品牌图标和主标题“整”均可进入活动创建页。

## 2. 前端页面与文件依赖

### 2.1 活动档案列表

涉及文件：

- `frontend/pages/activities/index.js`
- `frontend/pages/activities/index.wxml`
- `frontend/pages/activities/index.wxss`

数据流：

1. 页面 `onShow` 调用 `GET /api/activities/mine/archive`。
2. `mapArchiveRecord` 将后端字段转换为前端展示字段，例如 `statusLabel`、`roleLabel`、`totalAmountText`、`startTimeValue`。
3. `applyFilters` 在前端做组合筛选：身份、状态、起止日期、关键词、时间排序。
4. 点击卡片调用 `openArchiveDetail`，跳转到 `/pages/archive-detail/index?id=活动ID`。

筛选规则：

- 身份：`all`、`creator`、`member`。
- 状态：`all`、`finished`、`cancelled`、`active`。其中 `active` 表示既不是 `finished` 也不是 `cancelled`。
- 起止时间：按活动 `startTime` 过滤，开始日期取当天 00:00:00，结束日期取当天 23:59:59。
- 关键词：匹配活动名、类型、地点、亮点、身份和状态文案。

### 2.2 档案详情页

新增文件：

- `frontend/pages/archive-detail/index.js`
- `frontend/pages/archive-detail/index.wxml`
- `frontend/pages/archive-detail/index.wxss`
- `frontend/pages/archive-detail/index.json`

注册文件：

- `frontend/app.json`

数据流：

1. 页面从路由参数读取 `id`。
2. 并行请求：
   - `GET /api/activities/{id}` 获取活动详情、地点、成员、费用模式。
   - `GET /api/activities/{id}/expenses/summary` 获取账单总额、消费明细和结算结果。
3. `mapDetail` 负责活动字段格式化。
4. `mapBill` 负责金额格式化、消费笔数、人均参考和结算人员展示。

页面展示内容：

- 活动名称、状态、类型、线上/线下、时间、地点、到场人数、费用模式。
- 账单总览：总消费、人均参考、消费笔数、结算说明。
- 每一笔支出：事项、付款人、金额。
- 结算追溯：参与者、角色、应转金额。
- 参与成员列表。

### 2.3 AI 人格档案

涉及文件：

- `frontend/pages/personality/index.js`
- `frontend/pages/personality/index.wxml`
- `frontend/pages/personality/index.wxss`

数据流：

1. 页面调用 `GET /api/activities/mine/personality-report`。
2. `mapReport` 对后端结果进行前端适配：
   - 为六维指标补 `axisClass` 和百分比文案。
   - 根据六个百分比生成 CSS `clip-path: polygon(...)`，形成不规则雷达内部多边形。
   - 为旧接口或空数据补空财报对象，避免前端空引用。
3. 点击“热力”按钮触发 `toggleHeatmap`，展开或收起六维雷达图。
4. 财报切换使用 `selectFinancePeriod`，从 `financeReport.daily/weekly/monthly/quarterly/yearly` 中取当前维度数据。

六维指标：

- 组局力：最近窗口内发起活动数。
- 参与度：最近窗口内参与别人活动数。
- 带动力：到场人数贡献，按每场 `joinedCount - 1` 累加。
- 落地率：最近窗口内已结束活动占比。
- 夜场热度：晚上 22 点后或凌晨 5 点前活动占比。
- 连续性：连续活跃天数。

## 3. 后端接口与文件依赖

### 3.1 已复用接口

控制器：`backend/src/main/java/com/playminipro/activity/controller/ActivityController.java`

- `GET /api/activities/mine/archive`：活动档案列表。
- `GET /api/activities/{id}`：活动详情。
- `GET /api/activities/{id}/expenses/summary`：活动账单汇总。
- `GET /api/activities/mine/personality-report`：人格档案和财报。

### 3.2 新增 DTO

新增文件：

- `ActivityFinanceRowResponse`：Mapper 查询出的个人财务原始行。
- `PersonalityRadarMetricResponse`：人格六维指标。
- `PersonalityFinanceBucketResponse`：某个日/周/月/季/年桶内的花费。
- `PersonalityFinanceReportResponse`：人格页总财报。

修改文件：

- `PersonalityReportResponse` 增加 `radarMetrics` 和 `financeReport`。

### 3.3 后端聚合逻辑

核心服务：`backend/src/main/java/com/playminipro/activity/service/ActivityInsightService.java`

人格报告数据源：

1. `listMineArchive(userId)`：用于最近 30 天人格统计、DNA、称号、雷达指标。
2. `activityMapper.findMineFinanceRows(userId)`：用于全量个人财报。

财务口径：

- 取消活动不计入财报。
- `aa`：用户花费 = 活动总支出 / 到场人数。
- `host_treat`：如果当前用户是发起人，则用户花费 = 活动总支出；如果是参与者，则为 0。
- `designated_treat`：当前阶段先按发起人承担整单处理。后续如果补指定付款人字段，需要改这里的计算口径。
- `none`：不计入花费。

分桶维度：

- 日：`yyyy-MM-dd`，最多返回最近 14 个有支出的日期。
- 周：`yyyy-Wxx`，ISO 周，最多返回最近 12 周。
- 月：`yyyy-MM`，最多返回最近 12 月。
- 季：`yyyy-Qx`，最多返回最近 8 季。
- 年：`yyyy`，最多返回最近 6 年。

性能设计：

- 后端不为每个维度分别查询数据库，而是通过 `findMineFinanceRows` 一次查出该用户参与过的非取消活动及活动总支出。
- Java 内存中做五种维度分桶，避免日/周/月/季/年五次 SQL 聚合。
- 查询依赖现有索引：`activity_members(user_id, join_status)`、`activity_members(activity_id, join_status)`、`activity_expenses(activity_id, created_at DESC)`。
- 当前返回每个维度只保留最近若干桶，避免长期用户一次返回过多历史数据。

## 4. 数据表说明

本次功能涉及表：

- `users`：用户昵称和头像，用于成员、付款人、人格报告昵称。
- `activities`：活动基础信息、状态、时间、费用模式。
- `activity_members`：用户和活动的关系、角色、加入状态、加入时间。
- `activity_expenses`：每笔消费，包含活动 ID、付款人、消费事项、金额。

关键字段：

- `activities.status`：用于档案状态筛选、财报排除取消活动。
- `activities.start_time`：用于档案起止日期筛选和财报分桶。
- `activities.expense_mode`：决定 AA、请客、无需结算的财务口径。
- `activity_members.role`：决定“我发起的 / 我参加的”和请客统计。
- `activity_expenses.amount_fen`：所有金额以分为单位，前端展示时格式化为元。

## 5. 技术与工具

- 前端：微信小程序原生 WXML / WXSS / JS。
- 后端：Spring Boot 3.3、Java 21、MyBatis 注解 SQL。
- 数据库：PostgreSQL。
- 迁移工具：Flyway。
- 中间件：Redis 仍用于既有登录会话校验，本次财报不新增缓存依赖。
- 算法：
  - 六维雷达：后端生成六个 0-100 分指标，前端按 60 度间隔换算为 CSS polygon 点。
  - 财报分桶：按活动开始日期映射到日/ISO 周/月/季/年，然后按桶累计金额和活动数。

## 6. 调试建议

### 6.1 活动档案详情

1. 打开活动档案页，验证身份、状态、开始日期、结束日期和关键词是否可组合筛选。
2. 点击任意历史活动卡片，应进入“档案详情”页面，而不是底部弹窗。
3. 如果详情页账单为空，先去账单页为对应活动记一笔消费，再回到档案详情刷新。
4. 如果接口返回 401，确认本地登录态是否过期。

### 6.2 人格雷达图

1. 进入 AI 人格档案页。
2. 点击“热力”圆环，应展开六维雷达图。
3. 如果雷达图未显示，优先检查接口返回是否包含 `radarMetrics`，再检查 `report.radarPolygon` 是否是合法 polygon 字符串。

### 6.3 财报

1. 检查人格页“我的财报”的总花费是否等于请客花费 + AA 花费。
2. 切换日/周/月/季/年，确认列表随 `financePeriod` 切换。
3. 对 AA 活动，用活动总金额除以到场人数验证单个用户花费。
4. 对请客活动，确认只有发起人的财报计入整单金额。

## 7. 后续可人工扩展点

- 指定请客：当前 `designated_treat` 按发起人承担整单计算；后续可在 `activity_expenses` 或结算表中增加“承担人/被请人”字段，再修改 `calculateUserSpentFen`。
- 财报缓存：当活动和账单数量明显增加时，可以增加按用户和月份聚合的缓存表或 Redis 缓存。
- 图表增强：当前雷达图使用 WXSS polygon 实现；如果后续需要动画、截图或更精细标尺，可改为 Canvas 组件。
- 后端筛选下推：活动档案目前在前端组合筛选。用户数据量变大后，可把状态、起止时间、身份、关键词下推到 `GET /api/activities/mine/archive` 查询参数。

# 202606021041 邀请闭环、成员通知、孤活动自动取消工程设计与调试说明

## 1. 本次改进范围

本次改进围绕活动分享后的完整闭环：

- 好友从活动详情页点击“+”分享进入小程序后，登录完成会回到被分享的活动详情页，而不是停留在首页。
- 被邀请用户能在详情页查看活动信息，选择“接受邀请”或“婉拒”。接受后写入活动成员，详情页头像和成员列表立即刷新。
- 活动内已加入成员通过后端通知事件表收到“新成员加入”活动通知事件。当前版本先落库，方便人工调试；接入微信订阅消息模板后，在同一服务里补真实发送。
- 活动开始时间超过 2.5 小时且仍只有发起人时，后端自动生成“半小时后自动取消”提醒事件。
- 活动开始时间超过 3 小时且仍只有发起人时，后端自动把活动状态改为 `cancelled`，并生成“已自动取消”通知事件。

## 2. 业务逻辑

### 2.1 分享进入活动详情

1. 活动详情页分享路径为 `/pages/detail/index?id=活动ID&source=invite`。
2. 好友打开详情页时，如果本地没有登录态，详情页把完整目标路径写入本地缓存 `pendingInvitePath`。
3. 详情页跳转到首页登录面板。
4. 首页完成静默登录或手动确认登录后，调用 `consumePendingInvitePath` 读取并删除 `pendingInvitePath`，再 `wx.navigateTo` 回到活动详情。
5. 详情页调用 `GET /api/activities/{id}` 获取活动详情。后端现在允许非成员查看活动基础信息和已加入成员，用 `canJoin` 告诉前端是否展示接受/婉拒按钮。

### 2.2 接受邀请

1. 前端点击“接受邀请”时，先调用 `requestInitialSubscribePermission` 请求微信订阅消息授权。
2. 前端调用 `POST /api/activities/{id}/join`。
3. 后端校验活动存在、状态可加入、活动未满员。
4. 后端使用 `activity_members(activity_id, user_id)` 唯一约束做幂等加入：
   - 从未加入过：插入 `role = member`、`join_status = joined`。
   - 之前婉拒或退出：更新为 `join_status = joined`，刷新 `joined_at`。
   - 已加入：不重复插入。
5. 后端重新统计 `joinedCount`，如果人数达到 `max_participant_count`，把活动状态从 `recruiting` 标记为 `full`。
6. 若本次确实发生新增或恢复加入，后端为除新成员外的所有已加入成员写入 `member_joined` 通知事件。
7. 前端重新调用详情接口，刷新头像席位和成员列表。

### 2.3 婉拒邀请

1. 前端点击“婉拒”时调用 `POST /api/activities/{id}/decline`。
2. 后端只允许非发起人成员关系被标记为 `quit`。
3. 如果用户之前没有成员关系，接口也返回成功，保证“婉拒”动作对用户是稳定的。
4. 前端提示“已婉拒”并回到首页。

### 2.4 成员实时可见

当前小程序端采用轻量轮询，详情页打开时每 10 秒刷新一次详情：

- 进入详情页：`startMemberRefresh` 启动定时器。
- 页面隐藏或卸载：`stopMemberRefresh` 清理定时器。
- 每轮刷新：调用 `GET /api/activities/{id}`，更新 `detail.members`、`joinedCount`、`canJoin`。

说明：这是当前项目依赖最少的实时方案，不新增 WebSocket 或消息队列。后续如果并发活动变多，可升级为 WebSocket、SSE，或将刷新间隔下发为配置。

### 2.5 孤活动自动提醒与取消

后端开启 Spring Scheduling，每 5 分钟扫描一次活动：

1. SQL 先筛选状态仍未结束/未取消、开始时间已超过 150 分钟、且 `joined` 成员数等于 1 的活动。
2. Java 判断当前时间是否已经超过活动开始时间 3 小时。
3. 未超过 3 小时：生成一次 `auto_cancel_reminder` 通知事件给发起人。
4. 已超过 3 小时：调用 `autoCancel` 把活动状态改为 `cancelled`，再生成 `auto_cancelled` 通知事件给发起人。
5. `auto_cancel_reminder` 通过 `existsForTarget` 去重，避免每次扫描重复提醒。

## 3. 接口清单

控制器文件：`backend/src/main/java/com/playminipro/activity/controller/ActivityController.java`

- `GET /api/activities/{id}`：获取活动详情。现在非成员也可以读取基础详情，用于分享预览与邀请决策。
- `POST /api/activities/{id}/join`：接受邀请并加入活动。
- `POST /api/activities/{id}/decline`：婉拒邀请。
- `POST /api/activities/{id}/cancel`：发起人主动取消，沿用原接口。

详情响应新增字段：

- `currentUserJoined`：当前用户是否已加入。
- `currentUserCreator`：当前用户是否是发起人。
- `canJoin`：当前用户是否可点击“接受邀请”。

接受邀请响应：

- `activityId`：活动 ID。
- `joined`：是否已处于加入状态。
- `joinedCount`：最新已加入人数。
- `maxParticipantCount`：最大人数。

## 4. 数据表与字段

### 4.1 已有表

- `users`：存用户昵称、头像、OpenID。成员通知内容会读取加入者昵称。
- `activities`：活动主体表，核心字段包括 `status`、`start_time`、`max_participant_count`、`creator_id`。
- `activity_members`：活动成员关系表，核心字段包括 `role`、`join_status`、`joined_at`。

`activity_members` 关键约束：

- `UNIQUE(activity_id, user_id)`：保证一个用户在一个活动里只有一条成员关系。
- `role IN ('creator', 'member')`。
- `join_status IN ('joined', 'quit', 'waiting')`。

### 4.2 新增表

迁移文件：`backend/src/main/resources/db/migration/V4__add_activity_notification_events.sql`

表名：`activity_notification_events`

用途：保存待发送、已发送或跳过的活动通知事件，便于人工调试和后续接入微信订阅消息。

字段说明：

- `id`：通知事件 ID。
- `activity_id`：所属活动。
- `target_user_id`：通知接收者。
- `actor_user_id`：触发动作的用户，例如新加入成员。自动取消类事件用发起人。
- `event_type`：事件类型，当前包括 `member_joined`、`auto_cancel_reminder`、`auto_cancelled`。
- `title`：通知标题。
- `content`：通知内容。
- `page_path`：用户点击通知后建议进入的小程序页面。
- `send_status`：发送状态，当前默认 `pending`。
- `payload`：预留 JSON 字段，后续可以保存微信接口返回或模板字段。
- `created_at`、`sent_at`：创建和发送时间。

索引：

- `idx_activity_notification_target_status`：按用户、发送状态、创建时间查询通知。
- `idx_activity_notification_activity_type`：按活动和事件类型排查通知。

## 7. 技术、中间件、工具与算法

- 前端：微信小程序原生 JS / WXML / WXSS。
- 后端：Spring Boot 3.3、Java 21、MyBatis 注解 SQL。
- 数据库：PostgreSQL。
- 数据迁移：Flyway。
- 定时任务：Spring Scheduling。
- 会话：沿用现有 JWT + Redis 登录态校验。
- 通知：当前版本使用 `activity_notification_events` 作为通知事件队列；后续接入微信订阅消息 API 时，不需要改业务入口，只需要增强 `ActivityNotificationService`。
- 实时成员可见算法：前端详情页固定间隔轮询，周期 10 秒。
- 自动取消算法：按活动开始时间做时间窗口判断，$T + 2.5h$ 提醒，$T + 3h$ 取消；数据库先过滤候选集合，Java 再判断具体动作。

## 8. 人工调试步骤

### 8.1 后端编译

在 `backend` 目录运行：

```powershell
mvn -q -DskipTests compile
```

### 8.2 数据库迁移

启动后端时 Flyway 会自动执行 `V4__add_activity_notification_events.sql`。

检查表是否创建：

```sql
SELECT to_regclass('public.activity_notification_events');
```

### 8.3 分享登录回跳

1. 用账号 A 创建活动。
2. 进入活动详情页，点击“+”分享给账号 B。
3. 账号 B 打开分享卡片。
4. 如果账号 B 未登录，应先进入首页登录面板。
5. 登录成功后，应自动跳回 `/pages/detail/index?id=活动ID&source=invite`。
6. 详情页应展示活动信息和“接受邀请 / 婉拒”。

### 8.4 接受邀请与成员头像

1. 账号 B 点击“接受邀请”。
2. 接口 `POST /api/activities/{id}/join` 应返回成功。
3. 详情页人数应从 `1 / N` 变为 `2 / N`。
4. 成员头像区应出现账号 B 的头像或昵称首字。
5. 账号 A 如果停留在详情页，最多 10 秒后也应看到成员列表刷新。

### 8.5 通知事件检查

账号 B 加入后，执行：

```sql
SELECT event_type, title, content, send_status, page_path, created_at
FROM activity_notification_events
WHERE activity_id = '活动UUID'
ORDER BY created_at DESC;
```

应该看到 `member_joined` 事件，接收者是已在活动里的其他成员。

### 8.6 婉拒邀请

1. 账号 B 打开分享详情。
2. 点击“婉拒”。
3. 页面应提示“已婉拒”并返回首页。
4. 如果账号 B 之前曾加入再婉拒，`activity_members.join_status` 应变为 `quit`。

### 8.7 自动提醒与取消

准备一个只有发起人的活动，把 `activities.start_time` 调到当前时间 2 小时 31 分钟前。

等待调度执行后检查：

```sql
SELECT event_type, content, send_status, created_at
FROM activity_notification_events
WHERE activity_id = '活动UUID'
ORDER BY created_at DESC;
```

应出现 `auto_cancel_reminder`。

再把 `start_time` 调到当前时间 3 小时 1 分钟前，等待调度执行后检查：

```sql
SELECT status
FROM activities
WHERE id = '活动UUID';
```

状态应变为 `cancelled`，通知事件里应出现 `auto_cancelled`。

## 9. 后续接入微信订阅消息的落点

当前已经有前端授权入口和后端通知事件落库。后续需要做三件事：

1. 在微信公众平台申请订阅消息模板，例如“成员加入提醒”“活动即将取消提醒”。
2. 把模板 ID 配置到前端 `subscribeTemplateIds` 或改成后端配置接口下发。
3. 在 `ActivityNotificationService` 中增加真实发送逻辑：读取 `users.open_id`、按 `event_type` 组装模板字段、调用微信 `subscribeMessage.send`，然后更新 `send_status` 为 `sent` 或 `failed`。

建议保留 `activity_notification_events`，即使真实发送成功也落库，便于排查“用户没收到通知”是未授权、微信接口失败，还是业务没有触发。

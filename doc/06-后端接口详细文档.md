# 朋友组局微信小程序后端接口详细文档

## 1. 文档目标

本文件用于把现有技术文档里的接口清单推进到可开发级别，重点覆盖：

1. 微信一键登录。
2. 活动创建、分享、回执、婉拒原因。
3. 费用记录与结算。
4. 用户画像与排行榜。

推荐后端实现栈：

1. Java 21。
2. Spring Boot 3。
3. Spring Web。
4. Spring Security + JWT。
5. MyBatis 或 MyBatis-Plus。
6. PostgreSQL。
7. Redis。

默认约定：

1. 接口前缀为 `/api`。
2. 鉴权方式为 `Authorization: Bearer <token>`。
3. 金额字段统一按分传输。
4. 时间统一使用 ISO 8601。
5. 主键统一使用 UUID 字符串。

## 3. 鉴权接口

### 3.1 微信一键登录

`POST /api/auth/wechat-login`

用途：

1. 接收小程序 `wx.login` 返回的 `code`。
2. 服务端调用微信 `code2Session`。
3. 创建或更新用户并签发业务 token。

请求体：

```json
{
  "code": "wx-login-code",
  "nickname": "阿强",
  "avatarUrl": "https://example.com/avatar.png"
}
```

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "token": "jwt-or-session-token",
    "user": {
      "id": "9db8d0c2-35f1-4620-9740-a278d91ceb2c",
      "nickname": "阿强",
      "avatarUrl": "https://example.com/avatar.png"
    },
    "isNewUser": true
  }
}
```

### 4.2 获取活动详情

`GET /api/activities/:id`

响应体建议包含：

1. 活动基础信息。
2. 当前席位信息。
3. 当前用户回执状态。
4. 回执聚合统计。
5. 费用摘要。

### 4.3 编辑活动

`PUT /api/activities/:id`

说明：

1. 只有发起人可调用。
2. 已开始活动建议限制部分字段不可改。

## 5. 回执接口

### 5.1 记录活动查看

`POST /api/activities/:id/view`

请求体：

```json
{
  "shareToken": "group_xxxxx",
  "sourceType": "share_link"
}
```

说明：

1. 用户点进活动页即可调用。
2. 同一用户重复查看不应重复累计唯一查看人数。

### 5.2 同意入局

`POST /api/activities/:id/rsvp/accept`

请求体：

```json
{
  "shareToken": "group_xxxxx"
}
```

响应体建议返回：

1. 最新回执状态。
2. 是否成功占位。
3. 当前剩余席位。

### 5.3 婉拒入局

`POST /api/activities/:id/rsvp/decline`

请求体：

```json
{
  "shareToken": "group_xxxxx",
  "reasonCode": "time_conflict",
  "reasonText": "今晚临时加班"
}
```

说明：

1. `reasonCode` 必填。
2. 当 `reasonCode = custom` 时，`reasonText` 必填。

### 5.4 重新考虑

`POST /api/activities/:id/rsvp/reopen`

用途：

1. 用户先婉拒，后续改主意。
2. 将状态重新置为 `viewed` 或待再次确认。

## 6. 成员接口

### 6.1 获取成员列表

`GET /api/activities/:id/members`

返回字段建议：

1. 用户信息。
2. role。
3. joinStatus。
4. seatIndex。
5. isWaitingList。

### 6.2 退出活动

`POST /api/activities/:id/quit`

说明：

1. 退出后应同步释放席位。
2. 若有候补，服务端可触发自动转正。

## 8. 结算接口

### 8.1 发起结算

`POST /api/activities/:id/settlements`

请求体：

```json
{
  "participantUserIds": [
    "u1",
    "u2",
    "u3",
    "u4"
  ]

```

响应体建议返回：

1. settlementId。
2. 总费用。
3. 人均承担。
4. 每人应收应付。
5. 最少转账路径。

`GET /api/activities/:id/settlements/latest`

`POST /api/settlements/:id/transfers/:transferId/pay`

### 8.4 确认我已收款

## 10. 错误码建议

建议至少统一这些错误码：

1. `4001` 活动不存在。
2. `4002` 活动已开始，禁止当前操作。
3. `4003` 活动人数已满。
4. `4004` 你不是该活动发起人。
5. `4005` 你不是该活动成员。
6. `4006` 当前回执状态不可重复提交。
7. `4007` 婉拒原因不合法。
8. `4008` 费用记录不存在。
9. `4009` 结算尚未生成。
10. `4010` 登录失效。

## 11. 开发实现建议

1. Java 后端建议使用 Spring Boot 3 + PostgreSQL，数据访问层优先使用 MyBatis 或 MyBatis-Plus。
2. 邀请回执和浏览记录不要只依赖前端状态，必须服务端落库。
3. 发起人查看实时反馈，首版可以先用短轮询，后续再接 WebSocket。
4. 所有金额统一按分存储，前端只负责格式化显示。
5. 排行榜和统计建议异步聚合，不要在高频页面做大聚合 SQL。
6. 涉及活动创建、入局、结算、回执状态流转的接口，建议放在 Service 层事务中处理。
7. 数据库版本变更建议统一使用 Flyway 管理，不直接手改线上库。

### 12.4 事务边界建议

以下操作建议强事务处理：

1. 创建活动并初始化发起人成员记录。
2. 同意入局并写入成员表、邀请表、事件表。
3. 婉拒入局并写入邀请表、事件表。
4. 发起结算并生成结算主表、结算明细、转账路径。

以下操作建议异步处理：

1. 用户统计聚合。
2. 排行榜快照刷新。
3. 通知推送。
4. 已查看未回复提醒。
